Most Watermill problems leave a clear trail in the logs. This page explains how to turn on verbose logging, how to isolate a single failing test case in the universal test suite, and how to diagnose deadlocks at runtime.

## Configuring logging

Watermill routes all internal log calls through the `LoggerAdapter` interface, which has four levels — Error, Info, Debug, and Trace. Debug and Trace are disabled by default because they are extremely verbose.

### LoggerAdapter interface

```go
type LoggerAdapter interface {
    Error(msg string, err error, fields LogFields)
    Info(msg string, fields LogFields)
    Debug(msg string, fields LogFields)
    Trace(msg string, fields LogFields)
    With(fields LogFields) LoggerAdapter
}
```

### StdLogger

`NewStdLogger` writes to stderr. Pass `true` for `debug` and `trace` to enable the verbose levels:

```go
// Signature: NewStdLogger(debug, trace bool) LoggerAdapter
logger := watermill.NewStdLogger(true, true)
```

You can also redirect output to any `io.Writer`:

```go
logger := watermill.NewStdLoggerWithOut(os.Stdout, true, true)
```

### slog adapter

`NewSlogLogger` wraps the standard library's structured logger. A `nil` argument falls back to `slog.Default()`:

```go
import "log/slog"

logger := watermill.NewSlogLogger(slog.Default())
```

Watermill adds a `Trace` level that `slog` does not have natively. It is mapped to `slog.LevelDebug - 4`. If you want to remap Watermill levels to different slog levels (for example, to silence noisy Info logs in production), use `NewSlogLoggerWithLevelMapping`:

```go
logger := watermill.NewSlogLoggerWithLevelMapping(
    slog.Default(),
    map[slog.Level]slog.Level{
        slog.LevelInfo: slog.LevelDebug, // demote Watermill info → slog debug
    },
)
```

### NopLogger

To silence all logging in tests or benchmarks:

```go
logger := watermill.NopLogger{}
```

## Debugging Pub/Sub tests

### Running a single scenario

```bash
go test -v ./... -run TestPublishSubscribe/TestContinueAfterSubscribeClose
```

### Grep by UUID

Every test execution in the universal test suite runs under a unique UUID sub-test:

```
--- PASS: TestPublishSubscribe (0.00s)
    --- PASS: TestPublishSubscribe/TestPublishSubscribe (2.38s)
        --- PASS: TestPublishSubscribe/TestPublishSubscribe/81eeb56c-3336-4eb9-a0ac-13abda6f38ff (2.38s)
```

That same UUID is used as the topic name in all messages for that run. You can redirect test output to a file and then grep for the UUID to see every log line belonging to a single test case:

```bash
go test -v ./... > test.out

# Find the UUID of the failing run in test.out, then:
cat test.out | grep 81eeb56c-3336-4eb9-a0ac-13abda6f38ff | less
```

This produces a focused timeline:


  <Accordion title="Common deadlock causes">
    The most frequent causes of deadlocks in Watermill applications:

    - **Publishing to a full channel with no subscriber**: `GoChannel` with `BlockPublishUntilSubscriberAck: true` will block `Publish` forever if there is no subscriber on the topic. Ensure you subscribe before publishing, or use the default non-blocking mode.
    - **Not calling Ack or Nack**: The subscriber output channel does not advance to the next message until the current one is Acked or Nacked. A handler that neither acknowledges nor returns will stall the entire subscriber.
    - **Closing while blocked**: `Close()` must unblock any in-progress `Publish` or `Subscribe` call. If your custom implementation holds locks during publish without checking the closing signal, `Close()` will deadlock.
  </Accordion>

  <Accordion title="Messages not being received">
    - Check that you are subscribing to the **same topic name** used for publishing. Topic names are case-sensitive strings.
    - If using `GoChannel` without `Persistent: true`, messages published before `Subscribe` is called are discarded. Subscribe first.
    - Verify that you are calling `msg.Ack()` after processing. Without Ack the channel blocks and no further messages are delivered.
    - Enable debug and trace logging (`NewStdLogger(true, true)`) to see whether messages are being published and consumed internally.
  </Accordion>

  <Accordion title="Messages delivered more than once">
    At-least-once delivery is the default. If you are seeing excessive duplicates:

    - Confirm your handler calls `Ack()` on success. If it panics or returns before acknowledging, the message is requeued.
    - Check whether your broker is configured with a short `ack deadline` / `visibility timeout`. Slow handlers that exceed the deadline cause the broker to redeliver.
    - For `ExactlyOnceDelivery`, set `Features.ExactlyOnceDelivery: true` in the test suite — this enables additional assertions that may reveal where duplicates originate.
  </Accordion>
</AccordionGroup>
