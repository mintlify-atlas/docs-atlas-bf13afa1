Watermill is designed to be broker-agnostic. Its core is built around two small interfaces — `Publisher` and `Subscriber` — so you can connect any messaging system by implementing them. This page walks through the contracts your implementation must satisfy, how to construct messages correctly, and how to validate your work with the universal test suite.

## The interfaces

Both interfaces live in `message/pubsub.go`. You must implement both to create a fully functional Pub/Sub backend.

```go
// Publisher is the emitting part of a Pub/Sub.
type Publisher interface {
    // Publish publishes provided messages to the given topic.
    //
    // Publish can be synchronous or asynchronous - it depends on the implementation.
    //
    // Most publisher implementations don't support atomic publishing of messages.
    // This means that if publishing one of the messages fails, the next messages will not be published.
    //
    // Publish does not work with a single Context.
    // Use the Context() method of each message instead.
    //
    // Publish must be thread safe.
    Publish(topic string, messages ...*Message) error
    // Close should flush unsent messages if publisher is async.
    Close() error
}

// Subscriber is the consuming part of the Pub/Sub.
type Subscriber interface {
    // Subscribe returns an output channel with messages from the provided topic.
    // The channel is closed after Close() is called on the subscriber.
    //
    // To receive the next message, `Ack()` must be called on the received message.
    // If message processing fails and the message should be redelivered `Nack()` should be called instead.
    //
    // When the provided ctx is canceled, the subscriber closes the subscription and the output channel.
    // The provided ctx is passed to all produced messages.
    // When Nack or Ack is called on the message, the context of the message is canceled.
    Subscribe(ctx context.Context, topic string) (<-chan *Message, error)
    // Close closes all subscriptions with their output channels and flushes offsets etc. when needed.
    Close() error
}
```

There is also an optional `SubscribeInitializer` interface for backends that require topic creation before consuming:

```go
// SubscribeInitializer is used to initialize subscribers.
type SubscribeInitializer interface {
    // SubscribeInitialize can be called to initialize subscribe before consume.
    // When calling Subscribe before Publish, SubscribeInitialize should be not required.
    SubscribeInitialize(topic string) error
}
```

## The Message struct

Messages are created with `message.NewMessage` and carry three public fields:

```go
type Message struct {
    // UUID is a unique identifier of the message.
    // It is only used by Watermill for debugging. UUID can be empty.
    UUID string

    // Metadata contains the message metadata.
    // Can be used to store data which doesn't require unmarshalling the entire payload.
    // It is something similar to HTTP request's headers.
    // Metadata is marshaled and will be saved to the PubSub.
    Metadata Metadata

    // Payload is the message's payload.
    Payload Payload
    // ...
}
```

Always construct messages with `NewMessage` rather than struct literals; the constructor initialises the internal `ack` and `noAck` channels:

```go
msg := message.NewMessage(watermill.NewUUID(), []byte("hello"))
msg.Metadata.Set("source", "my-service")
```

## Ack and Nack semantics

Your subscriber **must** block sending the next message on the output channel until the consumer has called `Ack()` or `Nack()` on the current message. The internal channels are exposed for exactly this purpose:

```go
select {
case <-msg.Acked():
    // consumer acknowledged — safe to commit offset / delete from queue
case <-msg.Nacked():
    // consumer rejected — redeliver the message
}
```

<Note>
`Ack()` and `Nack()` are idempotent and non-blocking. Calling either method closes the corresponding channel once, and subsequent calls are no-ops.
</Note>

## Implementation checklist

Before considering your Pub/Sub production-ready, work through this list:

<Steps>
  <Step title="Logging">
    Add structured log calls at appropriate levels (Error, Info, Debug, Trace) using the `watermill.LoggerAdapter` your constructor receives.
  </Step>
  <Step title="Configurable marshaller">
    Expose a replaceable message marshaller in your `Config` struct so users can swap serialisation formats.
  </Step>
  <Step title="Correct Close()">
    `Close()` must be idempotent, must not deadlock when a `Publish` or `Subscribe` is in progress, and must drain or close the subscriber output channel.
  </Step>
  <Step title="Ack and Nack">
    After a `Nack()` the message must be redelivered. After an `Ack()` it must not be redelivered.
  </Step>
  <Step title="Pass the universal test suite">
    Run `tests.TestPubSub` against your implementation (see below).
  </Step>
  <Step title="Performance">
    Profile and optimise before publishing — connection pooling, batching, and buffer sizes matter.
  </Step>
</Steps>

## The universal test suite

Watermill ships a test suite in `pubsub/tests` that every implementation should pass. It covers basic publish/subscribe, concurrent subscribers, error recovery, message ordering, context propagation, and reconnects.

### Key types

```go
// PubSubConstructor is a function that creates a Publisher and a Subscriber.
type PubSubConstructor func(t *testing.T) (message.Publisher, message.Subscriber)

// ConsumerGroupPubSubConstructor creates a Publisher and Subscriber using a given consumer group.
type ConsumerGroupPubSubConstructor func(t *testing.T, consumerGroup string) (message.Publisher, message.Subscriber)

// Features configures which capabilities your Pub/Sub supports.
type Features struct {
    ConsumerGroups                   bool
    ExactlyOnceDelivery              bool
    GuaranteedOrder                  bool
    GuaranteedOrderWithSingleSubscriber bool
    Persistent                       bool
    RestartServiceCommand            []string
    RequireSingleInstance            bool
    NewSubscriberReceivesOldMessages bool
    GenerateTopicFunc                func(tctx TestContext) string
    GenerateIDFunc                   func() TestID
    ForceShort                       bool
    ContextPreserved                 bool
}
```

### Running the suite

```go
package mypubsub_test

import (
    "testing"

    "github.com/ThreeDotsLabs/watermill/message"
    "github.com/ThreeDotsLabs/watermill/pubsub/tests"
)

func createMyPubSub(t *testing.T) (message.Publisher, message.Subscriber) {
    // construct your Publisher and Subscriber here
    pub := NewMyPublisher(...)
    sub := NewMySubscriber(...)
    return pub, sub
}

func TestMyPubSub(t *testing.T) {
    tests.TestPubSub(
        t,
        tests.Features{
            ConsumerGroups:      true,
            ExactlyOnceDelivery: false,
            GuaranteedOrder:     true,
            Persistent:          true,
        },
        createMyPubSub,
        nil, // provide a ConsumerGroupPubSubConstructor if ConsumerGroups is true
    )
}
```

Run with the race detector to catch concurrency issues:

```bash
go test -race ./...
```

### Stress testing

`TestPubSubStressTest` runs the full suite 10 times in parallel (configurable via `STRESS_TEST_COUNT`):

```go
func TestMyPubSubStress(t *testing.T) {
    tests.TestPubSubStressTest(
        t,
        tests.Features{
            Persistent: true,
        },
        createMyPubSub,
        nil,
    )
}
```

```bash
STRESS_TEST_COUNT=20 go test -race -run TestMyPubSubStress ./...
```
