Watermill makes it easy to build event-driven systems in Go. It gives you a single, consistent interface for publishing and consuming messages, regardless of which message broker or queue you're using underneath. Whether you're processing Kafka streams, routing RabbitMQ messages, or running a simple in-process pub/sub for tests, Watermill works the same way.

## Why Watermill?

Building distributed systems means dealing with message brokers — and every broker has its own client library, API, and quirks. Watermill hides that complexity behind a clean interface so you can focus on your application logic.

## How It Works

Watermill's architecture is built around three layers, each one building on the last:

<Steps>
  <Step title="Publisher & Subscriber">
    The lowest-level API. Publish messages to a topic and subscribe to receive them. Works with any supported backend.
  </Step>
  <Step title="Router">
    A higher-level construct that wires subscribers to handlers, manages goroutines, sends Acks/Nacks automatically, and lets you attach middleware for cross-cutting concerns.
  </Step>
  <Step title="CQRS Component">
    The highest-level API. Work with typed Go structs as events and commands instead of raw byte payloads. Ideal for event sourcing and domain-driven design.
  </Step>
</Steps>
